# Coding Conventions

## 언제

- 타입·파일·심볼 이름을 정하거나 새 타입을 선언할 때.
- 컴포넌트를 사용하거나 스타일을 적용할 때.
- 새 도메인 표준어를 정의하거나 주석·TSDoc을 작성할 때.
- 배치·패키지 규칙은 → [architecture](architecture.md).

## 규칙

- 포맷은 Prettier, 린트는 ESLint가 강제한다. 수동 스타일 논쟁을 하지 않는다 → [code-quality](code-quality.md).

## 네이밍

- 컴포넌트 파일·export는 PascalCase, 훅은 `use` 접두 camelCase, 그 외 모듈은 camelCase를 사용한다.
- 파일명은 역할을 드러낸다.
  - 컴포넌트: `Button.tsx`
  - 스토리: `Button.stories.tsx`
  - 훅: `useCart.ts`
  - Server Action: `actions.ts`
  - route handler: `route.ts`
  - 스키마: `schema.ts`
  - 슬라이스 public API: `index.client.ts` / `index.server.ts`
- 라우트 파일(`page`, `layout`, `loading`, `error` 등)은 `export default`를 사용한다.
  - 네임드 export는 라우터가 인식하지 못한다.
  - feature 컴포넌트는 네임드 export를 사용한다.
- 벤더 중립 명명을 사용한다.
  - 백엔드·인프라 벤더명을 타입·심볼에 넣지 않는다.
  - api-client 포트 명명은 → [data](data.md).
- 하나의 개념은 하나의 표준어만 사용한다.
  - 코드·UI 카피·타입명·API 필드가 같은 개념을 다르게 표현하지 않는다.
  - 도메인 표준어는 도메인 슬라이스 로컬 용어집에서 관리한다.

## 타입

- 백엔드 API 타입은 계약에서 생성한 Zod 스키마를 단일 원천으로 사용한다.
- 타입은 `z.infer`로 파생한다.
- 런타임 검증과 컴파일 타입 불일치를 막기 위해 타입과 스키마를 각각 선언하지 않는다.
- 경계 입력은 `unknown`으로 받고 Zod로 좁힌다.
- 검증 책임은 → [data](data.md)의 응답 형상·api-client에 따른다.
- TypeScript strict를 전제한다 → [code-quality](code-quality.md).
- `any`를 사용하지 않는다.

## 구조

- 명시성을 우선한다.
- 과한 추상화·배럴 남용·동적 재export를 피한다.
- 서버/클라 경계를 가로지르는 배럴은 금지한다.
  - 기준은 → [architecture](architecture.md)의 세그먼트와 서버/클라 구조 경계.

## 컴포넌트

### 배치·계층

- 컴포넌트 배치는 → [architecture](architecture.md)의 세그먼트와 서버/클라 구조 경계.
- 재사용 디자인시스템 컴포넌트는 `packages/ui`에 둔다.
- feature 전용 컴포넌트는 슬라이스 `ui`에 둔다.
- 도메인 로직을 `packages/ui`에 넣지 않는다.
- 계층·승격 기준·prop API·완료 정의는 → [design-system](design-system.md).

### 렌더링 경계

- 컴포넌트의 서버/클라 기본값과 `'use client'` 판단은 → [rendering](rendering.md).

### 접근성

- 접근성을 완료 조건으로 관리한다.
- 시맨틱 HTML을 우선한다.
- `div` 클릭 핸들러로 인터랙티브 요소를 대체하지 않는다.
- 인터랙티브 요소는 키보드 조작과 포커스 표시를 제공한다.
- 폼 입력은 연결된 `label`을 가진다.
- placeholder를 label 대용으로 사용하지 않는다.
- 정적 검사는 접근성 일부만 검증한다. 나머지 판단은 구현자가 소유한다.
  - 강제 범위는 → [code-quality](code-quality.md)의 ESLint.

### 리소스 최적화

- 이미지는 `next/image`를 사용한다.
  - dimensions를 명시한다.
  - LCP 이미지는 `priority`를 사용한다.
  - 컨테이너 채움은 `fill`을 사용한다.
- 폰트는 `next/font`를 사용한다.

## 스타일링

- Tailwind CSS v4 유틸리티와 디자인 토큰을 사용한다.
- 스타일 표현은 Tailwind CSS 유틸리티와 디자인 토큰 조합을 기본으로 한다.
- 원시 hex·px 대신 토큰을 사용한다.
- 원시값 강제는 → [code-quality](code-quality.md)의 ESLint.
- CSS-in-JS는 사용하지 않는다.
  - 런타임 비용 또는 스타일 관리 파편화를 방지하고 유틸리티 + 토큰 기반 일관성을 유지한다.
- 토큰 체계·명명·테마 분기는 → [design-system](design-system.md).
- 조건부 클래스는 명시적 유틸(`clsx`/`cn` 등)로 합성한다.

## 코드 주석·TSDoc

### 기본 규칙

- 주석은 한국어로 작성한다.
- 도메인 개념은 슬라이스 로컬 용어집의 표준어만 사용한다.
- 미등재 도메인 개념은 용어집에 먼저 등록한다.
- `shared`·`packages/ui`·`packages/config` 등 도메인 무지 계층의 기술 어휘는 제외한다.
- 도메인 개념이 타입 doc에 처음 등장하면 `한국어(영문식별자)`로 병기한다.
- 멤버 doc·라인 주석에는 병기하지 않는다.

## TSDoc

### 작성 대상

- 요약은 타입·함수·컴포넌트·훅·타입 멤버·유니온 리터럴에 작성한다.

#### 타입

- 타입 요약은 타입의 의미만 작성한다.
- 사용처·형상 이유는 문서가 소유한다.
- 멤버 의미는 멤버 doc이 소유한다.
- 타입 doc과 멤버 doc에 같은 내용을 반복하지 않는다.
- 이름만으로 의미가 명확한 타입은 doc을 생략할 수 있다.
- `z.infer` 파생 타입 별칭은 스키마 doc이 소유하므로 별도 요약을 두지 않는다.

#### 함수

- public API는 호출자 계약을 작성한다.
- 내부 함수는 담당 역할을 작성한다.
- 정상 호출자가 마주치는 전파 예외는 `@throws`로 작성한다.
- 선행조건 위반 예외는 계약에 포함하지 않는다.
- Server Action·route handler는 변경 내용과 무효화 캐시 태그를 작성한다.

#### 컴포넌트

- 렌더링하는 내용을 작성한다.
- 직접 fetch하거나 Suspense를 유발하면 작성한다.
- 실행 위치는 `'use client'`·`server-only` 가드가 소유하므로 반복하지 않는다.

#### 훅

- 반환값과 호출자가 지는 제약을 작성한다.
- 클라이언트 전용·마운트 이후 확정·정리 필요 여부를 포함한다.

#### 타입 멤버

- 값의 의미만 작성한다.
- 다음 정보를 포함할 수 있다.
  - 구분 기준
  - 단위·스케일·시간대
  - 값 범위
  - 존재 조건
  - 다른 멤버와의 산식
- 해당 값이 유발하는 동작·렌더 분기는 작성하지 않는다.
- `Zod .describe()`로 멤버 doc을 대체하지 않는다.

#### 유니온 리터럴

- 의미와 상태 진입 조건을 작성한다.
- 멤버당 한 줄을 넘기지 않는다.
- 전이 그래프와 나가는 전이는 백엔드가 소유한다.

### 제외 대상

- 라우트 파일(`page`, `layout`, `loading`, `error`, `not-found`, `route`)은 파일명과 위치가 역할을 정의하므로 별도 요약을 두지 않는다.
  - 런타임·캐시·인가 전제 등 비자명한 계약만 작성한다.
- public API 배럴의 재export는 원본 doc이 소유한다.
- prop 전달만 하는 래퍼는 대상 doc이 소유한다.
  - 값 변형·합성·기본값 주입 시 doc을 작성한다.
- 테스트 케이스는 `describe`·`it` 설명 문자열을 사용한다.
- 스토리는 스토리 이름·args를 사용한다.
- 테스트·스토리의 목·픽스처·헬퍼 타입은 일반 규칙을 적용한다.

### 계약 기준

- TSDoc은 API 계약만 작성한다.
- 구현 방식(How)은 작성하지 않는다.
- 자신의 계약만 작성한다.
- 관측 가능한 계약(전파 예외·캐시 무효화·리다이렉트)은 협력자를 거쳐 실현돼도 작성한다.
- 협력자의 내부 동작과 계약은 반복하지 않는다.
- 공유 타입 doc에는 사용 화면·시점을 작성하지 않는다.

### 형식

- 계약·요약은 TSDoc 블록(`/** */`)에 작성한다.
- 구현 흐름·이유는 라인 주석(`//`)에 작성한다.
- 타입 요약은 계사문으로 작성한다.
- 함수·컴포넌트·훅 요약은 3인칭 서술형으로 작성한다.
- 멤버·리터럴 요약은 명사구로 작성한다.
- 모든 요약은 마침표로 끝낸다.
- 명령형과 장황한 주어형을 사용하지 않는다.
- 타입 연결은 `{@link}`를 사용한다.
- `@returns`와 자명한 `@param`은 작성하지 않는다.
- `@param`은 비자명한 제약에만 작성한다.
- 컴포넌트 prop 제약은 prop 타입 멤버 doc이 소유한다.

## 라인 주석

- 함수 단계와 코드만으로 알 수 없는 이유만 작성한다.

### 단계 표시

- 단계가 2개 이상이면 번호를 붙인다.
- 형식은 `// 1. 입력 검증`과 같은 명사구로 작성한다.
- 한 함수에는 한 단계 계층만 둔다.
- 하위 루프·콜백에는 중복 번호를 두지 않는다.

### 이유 설명

- 이유는 왜 이 순서인지, 왜 이 값인지, 왜 변경하지 않는지처럼 코드가 설명하지 못하는 내용을 작성한다.
- 이유는 독립된 줄로 작성한다.
- 단계 관련 이유는 단계 아래에 둔다.
- 시점 없는 사실로 작성한다.
- 이름이 표현하는 내용을 반복하지 않는다.
- 주석 없이는 이해하기 어려운 블록은 함수·컴포넌트 추출을 우선한다.

## 표기

- 코드·값은 백틱으로 표기한다.
- 타입·심볼은 `{@link}`로 표기한다.

## 금지 항목

- 결정 내러티브를 작성하지 않는다.
- 채택·폐기 이유는 문서에 작성한다.
- 타입·린트·포맷·경계 게이트가 검증하는 내용을 반복하지 않는다.
- 주석 처리된 코드를 남기지 않는다.
- 작성자·날짜를 기록하지 않는다.
- TODO는 이슈 번호를 포함한다.
