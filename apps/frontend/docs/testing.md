# Testing

## 언제

- 새 기능·버그 수정·리팩터의 완료 조건이 되는 테스트를 계획·작성·수정할 때.
- 검증 시나리오 목록을 만들고 합의받을 때.
- 시나리오를 어디까지 파생하고 테스트 추가를 어디서 멈출지 판정할 때.
- 테스트 레벨·백엔드 스텁 방식·픽스처 형태·테스트 도구를 결정할 때.
- 디자인시스템 컴포넌트 검증은 → [design-system](design-system.md)의 워크벤치.
- loading·error·empty 상태 파일의 생성 규칙은 → [rendering](rendering.md).
- loading·error·empty 상태의 검증은 이 문서를 따른다.
- 게이트 배선·워크벤치 게이트·실행 그래프는 → [code-quality](code-quality.md).

## 규칙

### 시나리오 소유

- 검증 시나리오 목록의 선정은 사람이 소유한다.
- 행동을 추가·변경하는 작업은 구현 전에 시나리오 목록을 한국어 문장으로 제시해 합의받는다.
- 코드와 테스트를 같은 주체가 쓰면 의도 해석 오류가 양쪽에 상관된다.
- 시나리오 문장 합의가 이 상관을 끊는다.
- 목록은 필수 시나리오 축에서 파생되는 케이스를 기본 포함한다.
- 합의 후 구현과 통과는 에이전트가 소유한다.
- 합의된 문장은 `it` 설명 문자열과 1:1로 대응시킨다.
- 버그 수정은 재현 시나리오가 곧 목록이다.
- 리팩터는 새 시나리오를 만들지 않는다.
- 전후 테스트 통과가 기준이다.
- 합의 없이 시나리오를 축소·삭제하지 않는다.

### 작업 절차

- 새 행동·버그 수정은 합의된 시나리오의 테스트를 구현·수정 전에 작성한다.
- 타입·임포트에 필요한 최소 시그니처만 먼저 만든다.
- 구현 본문은 채우지 않는다.
- 테스트를 실행해 시나리오마다 기대한 단언 실패로 실패함을 확인한다.
- 컴파일 오류·모듈 해석 오류·셋업 오류는 실패 확인으로 치지 않는다.
- 구현 전에 통과한 신규 시나리오 테스트는 단언이 행동을 구분하는지 재검토한다.
- 실패 확인 후 구현해 전체를 통과시킨다.
- 이미 존재하는 행동의 시나리오 보강 테스트는 즉시 통과가 정상이다.
- 사전 실패 확인을 요구하지 않는다.
- 리팩터는 이 절차를 따르지 않는다.
- 기준은 시나리오 소유의 리팩터 규칙을 따른다.

### 테스트 레벨

- 행동마다 검증 가능한 가장 낮은 레벨 하나를 고른다.
- 같은 행동을 상위 레벨에서 반복하지 않는다.

| 레벨 | 대상 | 환경 | 선택 기준 |
|---|---|---|---|
| 단위 | 순수 함수·뷰 파생·수기 Zod 스키마·훅 로직 | Vitest(node) | DOM 없이 검증 가능한 행동 |
| 컴포넌트 | feature 슬라이스 ui·도메인 UI | Vitest + Testing Library(jsdom) | 상호작용·조건 렌더가 행동 |
| 서버 로직 | route handler·Server Action·api-client | Vitest(node) + MSW | 서버 경계 계약이 행동 |
| 워크벤치 | 디자인시스템 프리미티브·조합 | → [design-system](design-system.md)·[code-quality](code-quality.md) | — |
| E2E | 페이지 플로우·async RSC 렌더·스트리밍 | opt-in | 라우트 조립 자체가 검증 대상 |

- 디자인시스템 컴포넌트의 검증은 워크벤치 스토리가 소유한다.
- Testing Library 테스트를 중복 작성하지 않는다.
- async 서버 컴포넌트는 단위 러너로 렌더하지 않는다.
- 데이터 조립 로직은 함수로 추출해 단위·서버 로직 레벨에서 검증한다.
- 동기 서버 컴포넌트는 일반 함수 컴포넌트로 렌더해 검증할 수 있다.

### 배치·네이밍

- 테스트 파일은 `{대상}.test.ts`·`{대상}.test.tsx`로 대상 옆에 둔다.
- `__tests__` 디렉터리는 만들지 않는다.
- `describe`는 대상, 중첩 `describe`는 행동 그룹, `it`은 시나리오 하나를 서술한다.
- `it` 문자열은 조건과 결과를 담은 한국어 서술문으로 작성한다.
- 테스트 doc 문자열의 소유는 → [coding-conventions](coding-conventions.md)의 TSDoc 제외 대상을 따른다.
- 도메인 용어는 슬라이스 로컬 용어집의 표준어를 사용한다(→ [coding-conventions](coding-conventions.md)).

### 행동 명세

- 테스트는 사람이 코드 대신 읽는 행동 명세다.
- 테스트 하나는 행동 하나를 검증한다.
- 한 행동의 결과 단언 여러 개는 허용한다.
- 본문은 준비·실행·검증 3단을 빈 줄로 구분한다.
- given·when·then 라벨 주석은 두지 않는다.
- 라인 주석 규칙은 → [coding-conventions](coding-conventions.md).
- 단언 대상은 사용자 관측 결과(렌더 출력·응답·에러)다.
- 내부 상태·함수 호출 여부를 단언하지 않는다.
- 예외: 캐시 무효화처럼 부수효과 자체가 계약인 호출은 단언한다.
- 같은 행동의 값 경계 나열은 `it.each`로 표현한다.

### 필수 시나리오 축

- 아래 축에 해당하는 코드를 만들면 해당 시나리오를 목록에 포함한다.
- 축의 규칙 본문은 파생 원천 문서가 소유한다.

| 축 | 검증 시나리오 | 파생 원천 |
|---|---|---|
| async 상태 | loading·error·empty 각 상태의 렌더 | → [rendering](rendering.md) |
| ingress 검증 | 불량 입력의 거부, 표시 파라미터의 `.catch` 보정 | → [data](data.md) |
| 목록 egress | 봉투 실패 throw, 불량 항목 드롭과 로그 | → [data](data.md) |
| 상세 egress | 실패 시 타입드 에러 throw | → [data](data.md) |
| 에러 계약 | ProblemDetail의 타입드 에러 매핑, 미지 `code` 처리 | → [data](data.md) |
| 폼 | 필드 에러 표시, 서버 재검증의 거부 | → [data](data.md) |
| 인가 | Server Action·route handler 본문의 미인가 거부 | → [data](data.md) |
| 캐시 무효화 | 변경 성공 시 선언된 태그의 무효화 호출, 거부·실패 시 미호출 | → [data](data.md) |
| 변경 안전 | 낙관적 업데이트 실패 시 롤백, 변경 요청의 멱등키 포함 | → [data](data.md) |

### 시나리오 충분성

- 시나리오 목록은 아래 파생 규칙으로 제안한다.
- 최종 선정은 시나리오 소유를 따른다.
- 행동마다 대표 성공 시나리오 하나를 포함한다.
- 행동의 계약된 실패 경로(타입드 에러·ingress 거부·에러 상태 렌더)마다 실패 시나리오 하나를 포함한다.
- 계약된 실패 경로가 없는 행동은 성공 시나리오만 둔다.
- 코드가 유한하게 열거하는 계약은 전수 검증한다.
- 전수 열거는 `it.each` 하나로 묶는다.

| 유한 계약 | 전수 범위 | 레벨 |
|---|---|---|
| 수기 Zod 스키마 | 제약마다 위반 값 하나, `.catch` 보정 파라미터는 보정 결과 | 단위 |
| 유니온 리터럴 상태 | 상태 값마다 렌더·분기 결과 | 단위·컴포넌트 |
| 에러 계약 | 매핑하는 ProblemDetail `code`마다 타입드 에러, 미지 `code` 폴백 | 서버 로직 |
| 인가 | Server Action·route handler마다 미인가 거부, 인가 성공 | 서버 로직 |

- 코드젠 생성 스키마·타입은 전수 대상이 아니다(→ 테스트하지 않는 것).
- variant·상태·테마 매트릭스는 워크벤치가 소유한다.
- 단위·컴포넌트 테스트로 반복하지 않는다(→ [design-system](design-system.md)).
- 열거 자체(리터럴 존재·값 나열)는 검증하지 않는다.
- 열거에 걸린 행동만 검증한다.
- 무한 입력은 계약이 구분하는 동등 클래스로 분할하고 클래스당 대표 하나를 검증한다.
- 인접 클래스의 경계는 경계값과 경계 밖 첫 값으로 검증한다.
- 경계값이 클래스 대표를 겸하면 별도 대표를 두지 않는다.
- 아래 표준 클래스는 계약이 해당 구분을 가질 때만 적용한다.
- strict 타입이 소거한 클래스는 테스트하지 않는다.
- 전 클래스 적용은 `unknown` ingress 경계에만 한다.

| 입력 | 표준 클래스 |
|---|---|
| 문자열 | 유효, 빈 문자열, 공백, 길이 경계 |
| 수치·시각 | 범위 경계, 경계 밖 첫 값 |
| 배열 | 빈, 1개, 복수 |
| 옵셔널 | `undefined`, `null`, 존재 |
| 기간·만료 | 경계 시각, 직후 |

- 시간 경계는 주입된 시각·fake timers로 검증한다(→ 결정성).
- 조합은 픽스처 기본값에서 요인 하나만 바꾼다.
- 픽스처가 조합의 기준점이다.
- 요인을 동시에 바꾸는 조합 시나리오는 규칙 문서·합의 목록에 상호작용 규칙이 명시된 경우에만 둔다.
- 경쟁 상태 재현 테스트를 두지 않는다.
- 중복 제출 계약은 변경 안전 축의 멱등키가 소유하고, 중복 처리 판정은 백엔드 소유다(→ [data](data.md)).
- 중지 규칙: 필수 시나리오 축·유한 계약·동등 클래스의 각 항목에 시나리오가 하나씩 대응하면 목록은 완결이다.
- 새 시나리오가 새 계약 항목·새 동등 클래스를 검증하지 않으면 추가하지 않는다.

### 컴포넌트 테스트

- Testing Library로 사용자 관점 행동을 검증한다.
- 쿼리는 역할·접근명(`getByRole`) 우선이다.
- `data-testid`는 최후수단이다.
- 상호작용은 `user-event`로 수행한다.
- 클라이언트 페치는 MSW가 응답한다.
- 훅·모듈 목으로 대체하지 않는다.
- 접근성 자동 검사 게이트는 워크벤치가 소유한다(→ [code-quality](code-quality.md)).
- 역할 기반 쿼리는 그 보완이다.

### 서버 로직 테스트

- Server Action·route handler는 함수로 직접 호출해 검증한다.
- 검증 대상은 ingress 거부, 인가 판정, 에러 매핑, 캐시 무효화 호출이다.
- `next/cache`·`next/navigation`은 공유 셋업 파일(`vitest.setup.ts`)이 목으로 제공한다.
- `redirect()`·`notFound()`는 throw로 단언한다.
- api-client의 egress 행동(봉투 throw·항목 드롭)은 MSW 응답으로 검증한다.

### 백엔드 스텁

- 백엔드 응답 대체는 MSW 네트워크 경계 한 곳에서 한다.
- `vi.mock`으로 api-client·데이터 훅을 모듈 목하지 않는다.
  - 모듈 목은 egress 검증·에러 매핑을 건너뛰어 계약 회귀를 놓친다.
- 핸들러 응답 픽스처는 스키마 `parse`를 통과시켜 만든다.
- 계약을 어긴 픽스처는 픽스처 생성 시점에 실패한다.
- `server-only`·`client-only` 포이즌 임포트는 테스트 셋업에서 무해화한다.
- 경계 강제는 빌드 게이트가 소유한다(→ [architecture](architecture.md)).

### 픽스처

- 픽스처는 유효한 기본값 객체를 만드는 팩토리 함수다.
- 시나리오가 바꾸는 값만 인자로 받는다.
- 픽스처는 사용하는 테스트 소스가 소유한다.
- 두 슬라이스 이상 쓰면 대상 read-model을 소유한 `entities` 슬라이스로 승격한다.
- 픽스처·헬퍼의 타입·doc 규칙은 → [coding-conventions](coding-conventions.md).

### 결정성

- 테스트는 실행 순서에 독립이다.
- MSW 핸들러·스토어 상태는 테스트마다 초기화한다.
- 실패한 테스트를 재시도로 통과시키지 않는다.
- 비결정 원인을 제거한다.
- 타이머·디바운스는 fake timers로 제어한다.
- 시각·랜덤·로케일 비결정 값의 처리 규칙은 → [rendering](rendering.md)의 Hydration 결정성.
- 테스트도 같은 주입 경로를 사용한다.

### 테스트하지 않는 것

- 타입·린트·경계·워크벤치 게이트가 강제하는 내용은 반복하지 않는다(→ [code-quality](code-quality.md)).
- 코드젠 생성물(생성 스키마·타입)은 테스트하지 않는다.
- 계약 일치는 코드젠·타입 컴파일이 드러낸다(→ [data](data.md)).
- 라이브러리 자체의 동작(Zod·TanStack Query)은 테스트하지 않는다.
- 수기 스키마·쿼리 설계가 의도와 일치하는지는 테스트한다.
- 컴포넌트 스냅샷 테스트는 사용하지 않는다.
- 스냅샷은 행동 명세가 아니고 diff 승인이 형식화된다.
- 커버리지 수치는 게이트가 아니다.
- 합의된 시나리오 충족이 기준이다.
- 테스트만을 위한 프로덕션 공개 표면 확장을 금지한다.
- 실패하는 테스트를 통과시키기 위해 단언을 약화하거나 삭제하지 않는다.
- 구현을 고쳐 통과시킨다.
- 스킵 테스트는 이슈 번호 TODO와 함께만 둔다(→ [coding-conventions](coding-conventions.md)의 금지 항목).

### 도구

- 테스트 도구는 아래로 고정한다.
- 버전 정본은 각 프로젝트 `package.json`·`pnpm-lock.yaml`이다.

| 역할 | 채택 |
|---|---|
| 러너 | Vitest |
| DOM 환경 | jsdom |
| 컴포넌트 | Testing Library + `user-event` |
| 네트워크 스텁 | MSW |
| E2E | Playwright (opt-in) |

- Jest는 사용하지 않는다.
  - ESM·TypeScript 실행에 변환 계층 설정이 필요하다.
  - 워크스페이스 패키지의 TS 소스 직접 소비가 패키지별 transform 설정을 요구한다.
  - Vitest는 Vite 파이프라인으로 이를 기본 처리하고, API가 Jest 호환이라 전환 비용이 낮다.
- happy-dom은 사용하지 않는다.
- 선택 기준은 실행 성능이 아니라 브라우저 호환 충실도다.
- E2E는 baseline이 아니다.
- 도입 시 Playwright로 이 문서에서 고정한다.
  - async RSC·스트리밍·라우트 조립 검증은 E2E 소관이다.
  - opt-in 전까지 이 영역은 로직 추출 단위 검증으로 대체한다.
- 테스트 태스크는 turbo 그래프에 등록한다.
- 로컬·CI 동일 원칙은 → [code-quality](code-quality.md).

### 완료 판정

- 에이전트는 테스트 작성 후 이 목록으로 자기검증한다.
- 합의된 시나리오마다 대응하는 `it` 테스트가 있다.
- 새 행동·버그 재현 테스트는 작업 절차대로 구현·수정 전에 실패를 확인했다.
- 행동마다 성공 시나리오와 계약된 실패 경로 시나리오가 있다.
- 시나리오 충분성의 유한 계약·동등 클래스 항목마다 대응 테스트가 있다.
- 테스트 태스크와 게이트가 통과한다(→ [code-quality](code-quality.md)).
- 테스트를 위한 프로덕션 변경이 없다.
- 합의 없이 축소·삭제·스킵된 테스트가 없다.
