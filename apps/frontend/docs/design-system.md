# Design System

## 언제

- 디자인 토큰 추가·변경, 새 색·치수·모션 값 도입 시.
- 재사용 UI 컴포넌트 생성, feature UI의 디자인시스템 승격 판단 시.
- 컴포넌트 prop API(variant·size·className) 설계 시.
- 디자인시스템 컴포넌트 완료 기준 확인 시.
- 표기·명명·접근성 기준은 → [coding-conventions](coding-conventions.md)을 따른다.
- 배치·의존 방향은 → [architecture](architecture.md)를 따른다.
- 서버/클라이언트 렌더 경계는 → [rendering](rendering.md)을 따른다.

## 규칙

- 디자인시스템은 토큰 → 프리미티브 → 조합 → 도메인 UI의 단방향 계층이다.
- 하위 계층은 상위 계층과 도메인을 알지 못한다.
- 이 문서는 체계와 판단 기준만 소유한다.
- 구체 값 정의는 토큰 정의 스타일시트가 소유한다.
- 스타일 표기와 접근성은 → [coding-conventions](coding-conventions.md)가 소유한다.
- 강제 도구는 → [code-quality](code-quality.md)가 소유한다.

### 토큰

- 토큰은 원시값과 시맨틱 토큰의 2계층으로 구성한다.
- 원시값(hex·px·ms)은 토큰 정의 스타일시트에만 둔다.
- 코드는 역할 기반 시맨틱 토큰만 사용한다.
- 컴포넌트 스타일에 직접 색·치수 값을 선언하지 않는다.
- 시맨틱 토큰 이름은 역할을 표현한다.
- 색상명·수치명 기반 이름은 노출하지 않는다.

| 분류 | 정의하는 것 |
|---|---|
| 색 | 캔버스·표면·경계·내용·강조·상태 역할 색 |
| 타이포 | 제목·본문·캡션 등 역할 단위 크기·행간·굵기 |
| 스페이싱 | 패딩·갭 단계 |
| radius·shadow | 모서리·입면 단계 |
| 모션 | duration·easing 단계 |
| 레이어 | 겹침 순서 단계 |
| 브레이크포인트 | 반응형 분기 단계 |

- 새 값이 필요하면 기존 토큰으로 표현 가능한지 먼저 확인한다.
- 새 토큰은 새로운 역할이 생긴 경우에만 추가한다.
- 타이포는 역할 토큰 단위로 사용한다.
- 크기·행간·굵기를 개별 조합해 역할을 재구성하지 않는다.
- 테마 분기는 토큰 정의 레이어가 소유한다.
- 컴포넌트에 테마 조건 분기를 두지 않는다.
- 토큰 정의 스타일시트는 디자인시스템 패키지가 소유한다.
- 앱별 토큰 재정의·복제는 금지한다.
- 토큰 값 선택 근거는 토큰 정의처 주석에 둔다.
- 이 문서는 구체 토큰 값을 정의하지 않는다.

### 컴포넌트 계층

- 컴포넌트는 프리미티브, 조합, 도메인 UI로 구분한다.
- 도메인 어휘는 도메인 UI에만 둔다.

| 계층 | 담는 것 | 위치 |
|---|---|---|
| 프리미티브 | 단일 인터랙션·표시 요소 래핑 | 디자인시스템 패키지 |
| 조합 | 프리미티브·슬롯 기반 도메인 무지 패턴 | 디자인시스템 패키지 |
| 도메인 UI | 도메인 데이터·행위 결합 화면 조각 | feature 슬라이스 ui |

- 컴포넌트는 feature 슬라이스 ui에서 시작한다.
- 도메인 어휘 없이 표현 가능하고 두 번째 사용처가 생긴 경우 디자인시스템으로 승격한다.
- 단일 사용처 범용화는 금지한다.
- 컴포넌트 API는 두 번째 사용처에서 검증한다.
- 승격 시 도메인 이름을 역할 이름으로 변경한다.
- 도메인 이름이 남은 prop·variant는 승격 미완성이다.
- 조합 컴포넌트는 콘텐츠를 소유하지 않는다.
- 콘텐츠는 children 또는 명명 prop으로 받는다.
- 조합 컴포넌트에 도메인 카피·도메인 데이터를 내장하지 않는다.
- 아이콘은 단일 세트를 사용한다.
- 개별 SVG를 산발적으로 추가하지 않는다.

- 배치 규칙은 → [coding-conventions](coding-conventions.md)의 컴포넌트를 따른다.
- 의존 방향 강제는 → [architecture](architecture.md)의 빌드가 강제하는 불변식을 따른다.

### 컴포넌트 API

- 비주얼 변형은 유한한 `variant`·`size` prop으로 정의한다.
- 색·패딩 등 임의 스타일 값을 받는 prop은 만들지 않는다.
- variant 이름은 역할 기반으로 정의한다.
- 색 이름·도메인 이름 기반 variant는 금지한다.
- 상호 배타 상태는 boolean prop 대신 열거형 prop으로 표현한다.
- `className`은 외부 배치 변경 용도로만 허용한다.
- `className` 허용 범위는 margin·grid 배치·너비 등 외부 조정에 한정한다.
- 내부 패딩·색·타이포 변경 용도의 `className` 사용은 금지한다.
- 반복되는 내부 스타일 변경은 variant로 승격한다.
- variant로 표현할 수 없는 반복 변경은 별도 컴포넌트로 분리한다.
- 래핑한 네이티브 요소의 속성과 이벤트는 전달한다.
- 시맨틱 요소를 유지한다.
- hover·focus-visible·disabled 등 인터랙션 상태는 CSS 의사 클래스로 처리한다.
- JS 상태로 인터랙션 상태를 복제하지 않는다.

### 완료 정의

- 디자인시스템 컴포넌트는 아래 조건을 모두 충족한다.
- 상태를 모두 정의한다.
  - 기본
  - hover
  - focus-visible
  - disabled
  - 해당 시 loading·에러·빈 상태

- 라이트·다크 테마 모두에서 검증한다.
- 컴포넌트 내부에 테마 분기를 두지 않는다.
- 접근성 완료 조건을 충족한다(→ [coding-conventions](coding-conventions.md)의 접근성).
- 모션 토큰을 사용한다.
- `prefers-reduced-motion`을 존중한다.
- 색·치수에 원시값을 사용하지 않는다.
- 워크벤치 스토리가 variant·상태 매트릭스를 검증한다.

### 워크벤치

- 디자인시스템 컴포넌트는 앱 외부 워크벤치에서 검증한다.
- 검증 범위는 variant × 상태 × 테마 매트릭스다.
- 스토리는 컴포넌트 사용 계약이다.
- 도메인 UI에는 스토리 작성을 강제하지 않는다.
- 컴포넌트·variant·상태 변경 시 스토리를 함께 변경한다.
- 접근성·비주얼 회귀 자동 검사는 워크벤치 스토리를 대상으로 연결한다.
- 워크벤치 도구·설정 위치·게이트 배선은 → [code-quality](code-quality.md)를 따른다.

### 변경과 폐기

- 토큰과 컴포넌트 prop API는 공개 계약이다.
- 이름 변경·제거·의미 변경은 전체 소비처 마이그레이션과 함께 처리한다.
- 레포 내부 소비처는 원자적으로 마이그레이션한다.
- 외부 소비처가 있으면 deprecate 기간을 거친다.
- 동일 역할의 토큰·컴포넌트 중복 생성은 금지한다.
- 기존 확장 또는 교체로 해결한다.
