# Design System

## 언제

- 디자인 토큰을 추가·변경하거나, 코드에 새 색·치수·모션 값이 필요할 때.
- 재사용 UI 컴포넌트를 만들거나, feature UI를 디자인시스템 패키지로 승격할지 판단할 때.
- 컴포넌트의 prop API(variant·size·className 경계)를 설계할 때.
- 디자인시스템 컴포넌트의 완료 기준을 확인할 때. 표기·명명·접근성은 → [coding-conventions](coding-conventions.md), 배치·의존 방향은 → [architecture](architecture.md), 서버/클라 판단은 → [rendering](rendering.md)의 렌더 경계.

## 규칙

- 디자인시스템은 토큰 → 프리미티브 → 조합 → 도메인 UI의 단방향 계층이다. 하위 계층은 상위 계층과 도메인을 모른다.
  - 일관성은 이 계층에서 나온다. 시각 결정이 아래 계층에 한 번 내려지면 위에서 반복되지 않는다.
- 이 문서는 체계·판단 기준만 소유한다. 구체 값의 정본은 토큰 정의 스타일시트, 스타일 표기·접근성은 → [coding-conventions](coding-conventions.md), 강제 도구는 → [code-quality](code-quality.md).

### 토큰

- 토큰은 2계층이다. 원시값(hex·px·ms)은 토큰 정의 스타일시트 안에만 살고, 코드는 시맨틱 토큰(역할 이름)만 소비한다(사용 규칙은 → [coding-conventions](coding-conventions.md)의 스타일링, 원시값 강제는 → [code-quality](code-quality.md)의 ESLint).
- 시맨틱 토큰은 역할로 명명한다. 원시값을 드러내는 이름(색상명·수치명)을 노출하지 않는다.
  - 역할 이름이어야 리브랜딩·테마 추가가 소비처 수정 없이 토큰 정의 교체로 끝난다.
- 표준 토큰 분류는 아래다. 새 역할이 이 분류로 표현되면 새 분류를 만들지 않는다.

  | 분류          | 정의하는 것                                                              |
  | ------------- | ------------------------------------------------------------------------ |
  | 색            | 캔버스·표면·경계·내용(주·보조·희미)·강조·상태(위험 등)의 역할 색          |
  | 타이포        | 제목·본문·캡션 등 롤 단위의 크기·행간·굵기 묶음                           |
  | 스페이싱      | 패딩·갭의 단계 스케일                                                     |
  | radius·shadow | 모서리·입면(elevation) 단계                                               |
  | 모션          | duration·easing 단계                                                      |
  | 레이어        | 겹침 순서(z-index) 단계 — 드롭다운·스티키·모달·토스트 등                  |
  | 브레이크포인트 | 반응형 분기 단계                                                          |

- 타이포는 롤 토큰 단위로 소비한다. 크기·행간·굵기를 유틸리티로 개별 조합해 롤을 재조립하지 않는다.
  - 롤을 우회하면 화면마다 미세하게 다른 텍스트 위계가 생긴다. 위계는 브랜드 인상에서 가장 먼저 무너지는 지점이다.
- 라이트·다크 등 테마 분기는 토큰 정의 레이어가 소유한다. 컴포넌트에 모드 조건(다크 전용 클래스 분기·JS 테마 분기)을 넣지 않는다.
  - 컴포넌트가 모드를 알면 분기가 모드 수 × 컴포넌트 수로 증식한다. 토큰 값 전환 한 곳이면 끝난다.
- 토큰 정의 스타일시트는 디자인시스템 패키지가 소유하고 각 앱이 import한다. 앱마다 재정의·복제하지 않는다.
  - 컴포넌트 패키지가 앱이 정의해 줄 토큰에 기대면 보이지 않는 계약이 생기고, 둘째 앱에서 복제되는 순간 드리프트한다.
- 토큰 값의 선택 근거(브랜드 팔레트·타이포 성격)는 토큰 정의처의 주석이 소유한다. 이 문서는 구체 값을 재서술하지 않는다.
- 새 시각 값이 필요하면 기존 토큰으로 표현되는지 먼저 본다. 토큰 추가는 새 역할이 생겼을 때만 한다.
  - 값이 다르다는 이유로 토큰을 늘리면 스케일이 원시값 목록으로 퇴화한다.

### 컴포넌트 계층

- 컴포넌트는 3계층으로 나눈다. 도메인 어휘는 도메인 UI에만 산다.

  | 계층       | 담는 것                                                                  | 위치                |
  | ---------- | ------------------------------------------------------------------------ | ------------------- |
  | 프리미티브 | 단일 인터랙션·표시 요소의 래핑(버튼·입력·배지 등)                         | 디자인시스템 패키지 |
  | 조합       | 프리미티브·슬롯을 합성한 도메인 무지 패턴(필드 그룹·카드 셸·빈 상태 등)   | 디자인시스템 패키지 |
  | 도메인 UI  | 도메인 데이터·행위와 결합된 화면 조각                                     | feature 슬라이스 ui |

  - 배치 규칙 자체는 → [coding-conventions](coding-conventions.md)의 컴포넌트, 의존 방향 강제는 → [architecture](architecture.md)의 빌드가 강제하는 불변식.

- 컴포넌트는 feature 슬라이스 ui에서 시작한다. 도메인 어휘 없이 표현할 수 있고 둘째 사용처가 생길 때 디자인시스템 패키지로 승격한다.
  - 사용처가 하나인 범용화는 투기다. 두 번째 사용처가 진짜 API를 알려준다.
  - 승격할 때 도메인 어휘를 역할 어휘로 치환한다. 도메인 이름이 남은 prop·variant는 승격 미완이다.
- 조합 컴포넌트는 콘텐츠를 소유하지 않는다. 콘텐츠는 슬롯(children·명명 prop)으로 받는다.
  - 콘텐츠·카피를 내장하면 도메인이 스며들어 도메인 무지가 깨진다.
- 아이콘은 단일 세트를 쓴다. 세트 밖 개별 SVG를 산발 도입하지 않는다.
  - 스트로크 굵기·모서리 처리 같은 시각 언어가 섞이면 브랜드 인상이 흐려진다.

### 컴포넌트 API

- 시각 변형은 유한한 variant·size prop으로 노출한다. 색·패딩을 임의 값으로 받는 자유 스타일 prop을 만들지 않는다.
  - 자유도가 무한하면 일관성이 사용처 단위로 무너진다. 변형이 유한해야 디자인 결정이 컴포넌트에 남는다.
- variant 이름은 역할(주·보조·위험 등)로 짓는다. 색 이름·도메인 이름으로 짓지 않는다.
- 서로 배타인 변형은 boolean prop 증식 대신 열거형 prop 하나로 받는다.
- className 병합은 바깥 배치(margin·grid 배치·너비)에 한정한 탈출구다. 내부 패딩·색·타이포를 사용처에서 덮지 않는다.
  - 내부를 사용처가 덮으면 그 스타일은 사실상 fork다. 같은 덮어쓰기가 반복되면 variant로 승격한다.
  - 조건부 클래스 합성 유틸은 → [coding-conventions](coding-conventions.md)의 스타일링.
- 래핑한 네이티브 요소의 속성·이벤트는 전달하고 시맨틱 요소를 유지한다(접근성 완료조건은 → [coding-conventions](coding-conventions.md)의 컴포넌트).
- 인터랙션 상태(hover·focus-visible·disabled)는 CSS 의사 클래스로 처리한다. JS 상태로 복제하지 않는다.

### 완료 정의

- 디자인시스템 컴포넌트는 아래를 모두 갖출 때 완료다. 코드 생성 후 이 목록으로 자기검증한다.
  - 상태 완비: 기본·hover·focus-visible·disabled, 해당하는 경우 loading·에러·빈 상태.
  - 라이트·다크 양 테마에서 확인. 컴포넌트 안에 모드 분기가 없어야 한다(위 토큰).
  - 접근성 완료조건 충족 → [coding-conventions](coding-conventions.md)의 컴포넌트.
  - 모션은 모션 토큰을 쓰고 모션 최소화 선호(prefers-reduced-motion)를 존중한다.
  - 원시값 없음: 색·치수는 시맨틱 토큰만 쓴다(위 토큰).
  - 워크벤치 스토리가 variant·상태 매트릭스를 커버한다(아래 워크벤치).

### 워크벤치

- 디자인시스템 컴포넌트는 앱 밖 워크벤치에서 variant × 상태 × 테마 매트릭스로 렌더해 검증한다. 스토리가 그 컴포넌트의 사용 계약이다.
  - 앱 화면만으로는 미사용 variant·다크 테마·엣지 상태가 검증에서 빠진다.
- 스토리 강제 범위는 디자인시스템 컴포넌트다. 도메인 UI에는 강제하지 않는다.
  - 도메인 UI는 도메인 데이터 목 유지비가 스토리의 계약 가치를 넘는다. 필요한 곳만 선택 도입한다.
- 컴포넌트·variant·상태를 추가하면 스토리를 같은 변경에서 추가한다.
  - 스토리가 뒤따르면 결국 안 쓰인다. 계약은 구현과 같은 커밋에 산다.
- 접근성·시각 회귀 자동 검사는 워크벤치 스토리를 대상으로 배선한다.
  - 사람 리뷰만으로는 토큰·공용 컴포넌트 변경이 일으키는 원거리 시각 회귀를 놓친다.
- 워크벤치 도구·설정 위치·게이트 배선은 → [code-quality](code-quality.md)의 Storybook.

### 변경과 폐기

- 토큰·컴포넌트 prop API는 소비처가 있는 공개 계약이다. 이름 변경·제거·의미 변경은 전 사용처 마이그레이션과 한 변경으로 묶는다.
  - 레포 안 소비처는 원자적으로 마이그레이션한다. 레포 밖 소비처가 생기면 deprecate 기간을 경유한다.
- 같은 역할의 토큰·컴포넌트를 중복으로 만들지 않는다. 기존 것을 확장하거나 교체한다.
  - 중복이 살아남으면 정본이 흐려지고 이후 변경이 반쪽만 적용된다.
